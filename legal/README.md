# @level9/brand/legal

Canonical source of truth for **legal attribution** and **privacy / terms / cookie policy templates** across every Level9 portfolio site.

One source → every site stays in sync. Change an LLC registered address here, bump the version, `npm install --force` in every consumer site, done.

Canonical map: [`attribution.ts`](./attribution.ts). Canonical entity records: [`entities.ts`](./entities.ts).

If you move a product between LLCs, **update `attribution.ts` first**, then re-run the site cleaner against that site so the footer + policy pages refresh.

## Mix model: external compliance + internal additions

Each policy is scaffolded with two kinds of sections, visually distinguished by a small badge in the rendered doc:

| Badge | Source | Who writes it |
|---|---|---|
| **Iubenda** (green) | External compliance generator | Generated by Iubenda per LLC's registered data practices. Paste output into `<PasteExternalHere source="iubenda" />` blocks. |
| **Termly** (blue) | External compliance generator | Same pattern, for Terms-side boilerplate (disclaimers, liability caps). |
| **Internal** (violet) | Product-specific knowledge | Things Iubenda doesn't know: our exact processor list, retention timelines, product-specific data flows, consulting engagement structure. |
| **Shared** (grey) | Generic legal boilerplate | "What are cookies," "Changes to policy," contact — edited once, shared across templates. |

**Why this mix:** Iubenda/Termly stay current on jurisdiction-specific requirements (GDPR updates, CPRA amendments). You handle product-specific facts (which AI provider you use, how long you keep AI outputs). No single generator gets both right.

## How consumer sites use these

### 1. Install / update the package

```bash
# In the consumer site's package.json, pin the branch or a tag:
"@level9/brand": "github:erichathaway/level9-brand#main"

# Then:
npm install --force
```

### 2. Wire the legal pages (Next.js App Router)

```tsx
// app/privacy/page.tsx
import { PrivacyPolicy } from "@level9/brand/legal/lucidorg";
export const metadata = { title: "Privacy Policy" };
export default function Page() { return <PrivacyPolicy />; }
```

```tsx
// app/terms/page.tsx
import { TermsOfService } from "@level9/brand/legal/lucidorg";
export const metadata = { title: "Terms of Service" };
export default function Page() { return <TermsOfService />; }
```

```tsx
// app/cookies/page.tsx
import { CookiePolicy } from "@level9/brand/legal/lucidorg";
export const metadata = { title: "Cookie Policy" };
export default function Page() { return <CookiePolicy />; }
```

### 3. Wire the footer

```tsx
// app/components/SiteFooter.tsx (or wherever your site's footer is)
import { LegalFooter } from "@level9/brand/legal";
export default function SiteFooter() {
  return <LegalFooter siteSlug="linkupos-site" />;
}
```

The footer resolves `siteSlug → entity` and renders the correct LLC copyright + legal-page links. Personal sites automatically omit `/terms` and `/cookies`.

### 4. Populate the `[PASTE IUBENDA OUTPUT HERE]` blocks

1. Generate the policy in Iubenda / Termly against the LLC's real registered data.
2. Open the relevant TSX file in `@level9/brand/legal/<llc>/` — find `<PasteExternalHere source="iubenda" />` blocks.
3. Replace each with the generated JSX / text.
4. Decrement `TEMPLATE_SECTIONS_REMAINING` at the top of the file as you go.
5. When zero, the "⚠ TEMPLATE" banner disappears and the doc is production-ready.

## Per-LLC doc status

Each LLC's template ships with a `templateSectionsRemaining` counter that renders a loud yellow "DO NOT DEPLOY YET" banner until every placeholder is replaced with real Iubenda/Termly output. The cleaner (see `site-cleaner.md` G1 + G3 gates) audits this counter and blocks production deploys when it's >0.

## Enforcement — site-cleaner audits this

The `site-cleaner` agent's Phase 1 audit checks for:

1. Does the site render `@level9/brand/legal/<correct-llc>/*` or a local copy?
2. Does the site's footer use `<LegalFooter siteSlug="…" />` with the right slug?
3. Is the LLC attribution in the footer the correct one per `attribution.ts`?
4. Are there zero `TEMPLATE_SECTIONS_REMAINING` left unresolved?

Findings flow into G1 (pitch 3 directions: adopt canonical / patch existing / rewrite) and G3 (sign-off by the `legal-business` + `privacy` officers before merge).

## When to update this package

Trigger a `@level9/brand` version bump + consumer `npm install --force` when:

- An LLC changes registered address, legal name, or governing jurisdiction (`entities.ts`)
- A product moves between LLCs (`attribution.ts`)
- You populate `<PasteExternalHere>` blocks for an LLC for the first time
- Compliance requirements change and all LLCs need a coordinated update

For per-site edits that should NOT propagate (e.g. linkupos-specific disclaimer), don't modify this package — add a local override in the consumer site's `app/privacy/page.tsx` after importing the canonical base.
